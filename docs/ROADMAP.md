# 路线图

## 阶段 1：网站 MVP

功能：

- 起卦流程。
- 卦象结果页。
- 事情类型选择。
- 基础白话解释。
- 八卦和六十四卦基础学习页。

技术方案：

- Next.js App Router。
- TypeScript。
- Tailwind CSS。
- Framer Motion。
- 数据先放在 `data/`，业务逻辑放在 `lib/`。

不要做：

- 不急着做原生 App。
- 不急着做小程序。
- 不引入复杂 CMS。
- 不做过度复杂的 3D 动画。

## 阶段 2：用户系统和历史记录

功能：

- 注册、登录。
- 用户资料。
- 历史占卜记录。
- 记录详情页。
- 基础后台查看。

技术方案：

- 先使用本地或轻量数据库结构验证。
- API 路由负责用户、记录和后台数据。
- 前端页面只展示和提交。

不要做：

- 不做复杂权限体系。
- 不做过早商业化闭环。
- 不把用户数据逻辑写死在页面组件里。

## 阶段 3：AI 解卦

功能：

- 根据用户问题、本卦、动爻、变卦整理事件脉络。
- 根据事情类型提取对应断语。
- 给出风险、机会、行动建议和复盘问题。
- 用户描述状态匹配卦象。

技术方案：

- 先构建稳定 prompt 模板。
- 服务端聚合卦象数据、用户问题和历史上下文。
- AI 输出要可追溯到本卦、动爻、变卦和原文依据。

不要做：

- 不让 AI 替代原文依据。
- 不输出绝对化结论。
- 不把模型调用散落在多个页面里。

## 阶段 4：PWA 移动网页

功能：

- 移动端体验强化。
- 离线或弱网基础访问。
- 添加到主屏幕。
- 更顺滑的移动端起卦动效。

技术方案：

- 优先优化现有 Web。
- 增加 PWA 配置。
- 检查移动端性能和可访问性。

不要做：

- 不为了 PWA 重写业务逻辑。
- 不引入沉重动画库。
- 不牺牲桌面端基础可用性。

## 阶段 5：iOS App

功能：

- 原生移动端起卦体验。
- 用户记录同步。
- 学习内容阅读。
- 重要提醒和复盘入口。

技术方案：

- 等 Web 数据结构稳定后再启动。
- 保持核心业务模型和接口可复用。
- 可评估 React Native、Expo 或原生 Swift 方案。

不要做：

- 不在网站 MVP 未稳定前开工。
- 不重复造一套完全不同的数据结构。
- 不让 App 端和 Web 端逻辑长期分叉。

## 阶段 6：微信小程序

功能：

- 轻量起卦。
- 查看结果。
- 历史记录。
- 学习入口。
- 分享和私域触达。

技术方案：

- 复用服务端 API 和数据结构。
- 小程序只做轻量展示与交互。
- 根据微信生态限制调整登录和支付。

不要做：

- 不把小程序做成完整后台。
- 不复制一套难维护的业务逻辑。
- 不在核心产品体验未稳定前过早投入。
